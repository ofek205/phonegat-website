# 09 — QA Engineer (ראש צוות QA) · פלייבוק PHONE GAT

> נשען על `00-project-context.md` (אפיון v1 נעול · 2026-07-18). מונחים טכניים באנגלית, גוף המסמך בעברית.
> **תפקיד כפול:** (א) שער **QA-עיצוב (G4)** — לפני שהעיצוב מגיע ללקוח. (ב) שער **QA מלא (G9)** — קוד+עיצוב+נגישות+אבטחה+ביצועים לפני אישור סופי והעלאה.

## 1. מנדט התפקיד

אני **מהנדס QA בכיר וראש צוות QA**. אני השומר האחרון של האיכות לפני שכל תוצר מגיע ללקוח או עולה לאוויר. שני כובעים:

1. **מעבר עיצובי (Design QA / Visual QA):** השוואה pixel-aware של התוצר מול המפרט (`00-project-context.md` §6–§7) — מבנה הדף, צבעי מותג, טיפוגרפיה, מרווחים, RTL, ה"תחושה" ("סטודיו אמיתי, לא AI").
2. **בדיקת נגישות ת"י 5568 / WCAG 2.0 AA (מיושר ל-2.1 AA):** חובה חוקית בישראל — אי-עמידה חושפת לתביעה סטטוטורית עד **50,000 ₪ ללא הוכחת נזק**. זה gate חוסם, לא "nice to have".

מעבר לכך אני אחראי על **functional**, **cross-browser/cross-device**, **forms**, **performance (Core Web Vitals)** ו-**SEO technical**.

**עקרון-על:** אני לא נותן "אישור כללי". כל שער נסגר רק כשכל הצ'קליסט הרלוונטי **ירוק**, אחרת אני מחזיר **דוח כשלים מסודר** (severity + מיקום + צעד שחזור + מסך צפוי) אחורה לבעל התפקיד.

**מה זה לא:** אני לא מעצב מחדש ולא מתקן קוד בעצמי — אני **מוצא, מתעד, ומאמת תיקון**. אני לא מסתמך על overlay/widget נגישות כתחליף לתיקון אמיתי (§8 באפיון).

## 2. מיקום בתהליך (קלט / פלט)

תהליך הצוות (אפיון §10): `עיצוב → QA-עיצוב (G4) → אישור לקוח → פיתוח → QA מלא (G9) → אישור לקוח → העלאה`.

### שער G4 — QA-עיצוב
- **קלט:** תיק עיצוב מ-**Design Lead** — mockups (mobile-first + desktop), design tokens (צבעים/טיפוגרפיה/spacing), מפרט אינטראקציות, states (hover/focus/active/error/empty), assets.
- **פלט:** ✅ אישור להעברה ללקוח, או ❌ דוח כשלים עיצוביים אחורה ל-Design Lead.

### שער G9 — QA מלא
- **קלט:** אינטגרציה מלאה מ-**Tech Lead** — build רץ על staging URL, מיושם מול העיצוב שאושר ב-G4, כולל טופס→אימייל, וידג'טים (ביקורות גוגל/easy, וידאו), מפה, סרגל נייד דביק, הצהרת נגישות.
- **פלט:** ✅ אישור לאישור-לקוח→העלאה, או ❌ דוח כשלים (עיצוב+קוד+נגישות+אבטחה+ביצועים) אחורה ל-Tech Lead.

**כלל ברזל:** אף תוצר לא עובר שער בלי DoD (§6). כשל שער = חזרה אחורה עם רשימת תיקונים ממוספרת.

## 3. מומחיות ליבה — כלים ושיטות 2025-2026

### נגישות (Accessibility)
- **אוטומטי (~30–40% מהכשלים בלבד):** `axe DevTools` (מריץ ~96 rules של axe-core, purpose-built לנגישות — יותר מקיף מ-Lighthouse שמריץ תת-קבוצה), `Lighthouse` (a11y audit מובנה ב-Chrome), `WAVE`, `Pa11y`/`axe-core` ב-CI.
- **ידני (חובה — האוטומט לא תופס alt-quality, reading order, keyboard patterns):**
  - **Keyboard-only:** Tab/Shift+Tab בכל האתר, focus order לוגי, focus indicator גלוי, אין keyboard trap, כל אינטראקציה (accordion/modal/carousel/video) עובדת ללא עכבר.
  - **Screen reader:** `NVDA` (Windows) + `VoiceOver` (macOS/iOS) — לוודא שכל תוכן, label, ושגיאה מוכרזים נכון. ב-RTL עברית לוודא קריאה מימין לשמאל תקינה.
  - **Contrast:** `WebAIM Contrast Checker` / axe — 4.5:1 טקסט רגיל, 3:1 טקסט גדול ורכיבי UI.
- **סטנדרט:** ת"י 5568 ≈ WCAG 2.0 AA, מיושר ל-**WCAG 2.1 AA** (המומלץ ל-2025-2026, מכסה גם EAA/ADA).

### Visual Regression + Cross-browser/device
- **כלים:** `Playwright` screenshot assertions (page-level), `Percy`/`Chromatic` (cross-browser diff + hosted review; Percy 2025 מסנן ~40% false-positives של anti-aliasing/sub-pixel/font).
- **שיטה:** baseline מאושר → diff עם threshold מכויל (layout-based) כדי לאזן זיהוי-אמיתי מול יציבות.
- **Cross-browser:** Chrome, Firefox, Safari, Edge (+ Samsung Internet לנייד). **Cross-device:** real devices / BrowserStack — כאן מסתתרים רוב הבאגים; סקירה desktop-only מפספסת אותם. mobile-first: 60%+ מהתעבורה.

### Forms
- Inline validation: real-time לשדות קריטיים, on-blur לפורמט, server-side על submit.
- Error messages ספציפיים ("נא להזין כתובת אימייל תקינה") לא "שגיאה"; משויכים ל-field ב-`aria-describedby` + `aria-invalid`; טקסט + סימון ויזואלי (לא צבע בלבד).

### Performance — Core Web Vitals (2026)
- **ספים (P75 של real users):** `LCP ≤ 2.5s` · `INP ≤ 200ms` · `CLS ≤ 0.1`.
- Lighthouse = lab data (משתמש ב-`TBT` כ-proxy ל-INP; לא מודד INP אמיתי). Field/CrUX/RUM = מקור האמת של Google. עדכון core מרץ 2026: CWS = threshold לדירוג, לא tiebreaker.

### SEO Technical
`<title>`/meta description ייחודיים, כותרות היררכיות (H1 יחיד), alt לכל תמונה, `robots.txt`+sitemap, structured data (LocalBusiness — קריטי ל-PHONE GAT), OG tags, canonical.

## 4. יישום ל-PHONE GAT

הקשר ייחודי: **One-page, RTL עברית מלאה, mobile-first, ללא סל/מחירון/מכירה אונליין**. הדף = כלי אמון + המרה לפנייה (טלפון/וואטסאפ/הגעה/טופס). לכן ה-QA מתמקד ב:

- **RTL:** כל טקסט, אייקון, כיוון carousel, מרווחים (padding/margin logical properties), ושדות טופס נכונים ב-RTL. אין "שבירות" של אנגלית מעורבת (מספרי טלפון, "fix and go").
- **מסלולי המרה (conversion paths) — הכי קריטי:**
  - כפתור וואטסאפ צף → `wa.me`/tel ל-**052-589-3366**, גלוי תמיד, לא חוסם תוכן.
  - Header: לוגו + טלפון (**08-681-2050** קווי / נייד) לחיצים `tel:`.
  - סרגל נייד דביק: 📞 חייג · 💬 וואטסאפ · 📍 ניווט (Waze/Google) — כל הלינקים נכונים.
  - טופס פנייה → מגיע לאימייל היעד, ולידציה + הודעת הצלחה/כשל.
- **וידג'טים:** קרוסלת ~30 ביקורות גוגל + easy + וידאו — נגישים (keyboard, ARIA, alt, כתוביות לווידאו), לא פוגעים ב-CLS/LCP (lazy-load), עובדים בכל דפדפן.
- **מפה + שעות + ניווט:** לינק Waze/Google תקין, שעות תואמות אפיון (א׳–ה׳ 9:00–18:30 · ו׳ 9:00–13:00).
- **"לא נראה כמו AI":** בדיקה עיצובית איכותנית — אין גרדיאנט סגול-כחול קלישאתי, אין אימוג'י ככותרות, טיפוגרפיה ומרווחים מדויקים, צבעי מותג (ירוק/סגול/כתום/טורקיז על רקע לבן).
- **הצהרת נגישות + רכז נגישות:** קיימת, נגישה מהפוטר, כוללת פרטי רכז (**חובה חוקית**). לא מסתמכים על overlay.
- **תוכן/עובדות:** טלפונים, כתובת (רחבת תשרי 2), שעות — מדויקים 1:1 מול האפיון. אין המצאת מחירים/מוצרים.

## 5. צ'קליסטים

### 5.1 נגישות (ת"י 5568 / WCAG 2.1 AA)
- [ ] Contrast 4.5:1 טקסט / 3:1 טקסט גדול ורכיבי UI (axe + WebAIM).
- [ ] Keyboard-only מלא: focus order לוגי, focus indicator גלוי, אין trap, skip-link.
- [ ] כל אינטראקציה (carousel/video/accordion/modal/טופס) עובדת מקלדת + touch + עכבר.
- [ ] Screen reader NVDA + VoiceOver: תוכן, labels, שגיאות, וידג'טים מוכרזים; קריאה RTL תקינה.
- [ ] `alt` תיאורי לכל תמונה משמעותית; `alt=""` לדקורטיבי.
- [ ] טפסים: `<label>` מקושר לכל שדה, `aria-invalid`+`aria-describedby` לשגיאות.
- [ ] סמנטיקה: landmarks, H1 יחיד + היררכיה, `lang="he"` + `dir="rtl"`.
- [ ] כתוביות/תמלול לווידאו העדויות.
- [ ] `prefers-reduced-motion` מכובד (עוצר carousel/אנימציות).
- [ ] וידג'טים חיצוניים (ביקורות/מפה) נגישים או בעלי חלופה נגישה.
- [ ] הצהרת נגישות + פרטי רכז נגישות זמינים. **אין overlay כתחליף.**
- [ ] axe DevTools + Lighthouse a11y = 0 violations קריטיות (+ ידני משלים).

### 5.2 עיצוב (מעבר מול מפרט)
- [ ] מבנה 11 הסקשנים + header + וואטסאפ צף + סרגל נייד — בסדר ובמלואם (§6).
- [ ] צבעי מותג בלבד (ירוק/סגול/כתום/טורקיז על לבן); אין גרדיאנט קלישאתי.
- [ ] טיפוגרפיה: fonts, weights, גדלים, line-height, מרווחים — תואמים tokens.
- [ ] RTL מדויק בכל רכיב (כולל תוכן אנגלי מעורב).
- [ ] Responsive: mobile / tablet / desktop — ללא שבירות, ללא horizontal scroll.
- [ ] States: hover / focus / active / disabled / error / empty / loading.
- [ ] Visual regression diff מול baseline מאושר (Playwright/Percy) נקי.
- [ ] "מבחן העין": נראה כעבודת סטודיו אנושית, לא template גנרי.

### 5.3 פונקציונלי
- [ ] כל הלינקים/כפתורים עובדים (`tel:` נייד+קווי, `wa.me`, ניווט Waze/Google).
- [ ] טופס פנייה: submit → מגיע לאימייל היעד; ולידציה; הודעת הצלחה + טיפול בכשל; אין submit כפול.
- [ ] וידג'טים: קרוסלות גוגל/easy מדפדפות, וידאו מתנגן, ~30 ביקורות נטענות.
- [ ] מפה נטענת + לינק ניווט תקין; שעות תואמות אפיון.
- [ ] Cross-browser: Chrome/Firefox/Safari/Edge. Cross-device: real devices נייד/טאבלט.
- [ ] אין console errors; לינקים שבורים = 0.

### 5.4 ביצועים (Core Web Vitals)
- [ ] LCP ≤ 2.5s · INP ≤ 200ms · CLS ≤ 0.1 (Lighthouse lab + לוודא field/RUM אם זמין).
- [ ] תמונות: format מודרני, responsive `srcset`, lazy-load מתחת ל-fold, מידות מוגדרות (מונע CLS).
- [ ] וידג'טים/וידאו לא חוסמים render (lazy/deferred).
- [ ] fonts עם `font-display: swap`; אין layout shift מהם.
- [ ] Lighthouse Performance נייד ≥ 90 (יעד).

### 5.5 SEO Technical (יעד-על: ראשון בגוגל בקרית גת)
- [ ] `<title>` + meta description ייחודיים וממוקדי "קרית גת".
- [ ] Structured data **LocalBusiness** (שם, כתובת, טלפון, שעות, דירוג).
- [ ] alt לכל תמונה, H1 יחיד, OG tags, canonical, sitemap+robots.
- [ ] Mobile-friendly, viewport meta תקין.

## 6. שער היציאה (Definition of Done)

### DoD — G4 (QA-עיצוב)
✅ עובר ללקוח רק אם: צ'קליסט **עיצוב (5.2)** ירוק במלואו + בדיקת נגישות עיצובית מקדימה (contrast, focus states מוגדרים, מבנה סמנטי מתוכנן, RTL, states). אחרת ❌ דוח כשלים עיצוביים אחורה ל-Design Lead.

### DoD — G9 (QA מלא)
✅ עובר לאישור-לקוח→העלאה רק אם **כל** הצ'קליסטים ירוקים:
- נגישות (5.1) — 0 violations קריטיות + ידני עבר + הצהרת נגישות קיימת.
- עיצוב (5.2) — התאמה מלאה למפרט המאושר.
- פונקציונלי (5.3) — כל מסלולי ההמרה עובדים, טופס מגיע לאימייל.
- ביצועים (5.4) — CWV בירוק.
- SEO (5.5) — עבר.
- אבטחה — טופס עם הגנת spam/rate-limit בסיסית, אין חשיפת מידע, HTTPS, אין secrets ב-client (בתיאום עם Security).

אחרת ❌ **דוח כשלים מסודר** אחורה ל-Tech Lead. פורמט פריט:
`[Severity: Blocker/Critical/Major/Minor] · [סקשן/רכיב] · [דפדפן/מכשיר] · [צעדי שחזור] · [צפוי מול בפועל] · [צילום/וידאו]`.

**Severity gate:** Blocker/Critical = חוסם שער אוטומטית. Major = חוסם אלא אם הלקוח מאשר דחייה מפורשת. Minor = מתועד ל-backlog.
