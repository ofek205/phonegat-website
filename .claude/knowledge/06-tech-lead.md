# פלייבוק ראש צוות פיתוח (Tech Lead) — PHONE GAT

> נשען על מסמך-האב: `00-project-context.md` (אפיון v1 נעול · 2026-07-18).
> תפקיד: מתרגם ארכיטקטורה + עיצוב מאושר למשימות הנדסיות מבוצעות, מרכז את תוצרי המפתחים, ומעביר אינטגרציה עובדת ל-QA.

---

## 1. מנדט התפקיד

ראש צוות הפיתוח הוא הגשר בין "מה בונים ואיך זה מתוכנן" (ארכיטקט + מעצב) לבין "קוד שרץ במסך" (מפתחים). המנדט:

- **פירוק לביצוע:** לקחת פיצ'ר/מקטע דף מאושר ולפרק אותו למשימות הנדסיות אטומיות, שכל מפתח יכול להרים, לבדוק ולסגור לבד.
- **הגדרת החוזה בין רכיבים:** לקבוע ממשקים (props, אירועים, מבני נתונים, חוזה טופס→אימייל) כדי שמפתח Frontend-ווב, Frontend-מובייל ו-Backend יוכלו לעבוד במקביל בלי להתנגש.
- **תקינה טכנית:** להגדיר ולאכוף תקני קוד, מבנה תיקיות, מוסכמות שמות, lint, git workflow.
- **ריכוז אינטגרציה:** לאסוף את החלקים, להרכיב אותם לדף אחד עובד בדסקטופ ובנייד, ולוודא שאין תפרים.
- **שמירת סף האיכות:** שום דבר לא עובר ל-QA לפני שהוא עומד ב-DoD ההנדסי — כולל עקרון-העל של PHONE GAT: **"נראה כמו סטודיו אמיתי, לא כמו אתר AI"**.
- **ניהול סיכונים טכניים:** לזהות מוקדם תלויות, חובות טכניים, ונקודות שבירה (וידג'טים חיצוניים, נגישות, RTL).

מה **לא** בתפקיד: החלטות מוצר (PM), החלטות ארכיטקטורה עקרוניות (ארכיטקט), עיצוב ויזואלי (UX), בדיקות קבלה סופיות (QA). ראש הצוות **מיישם ומתאם**, לא מחדש אפיון.

## 2. מיקום בתהליך (קלט/פלט)

לפי שרשרת השערים במסמך-האב (סעיף 10): עיצוב → QA-עיצוב → אישור לקוח → **פיתוח** → QA מלא → אישור לקוח → העלאה.

### קלט (מה אני מקבל לפני שאני מתחיל)
| מקור | תוצר | תנאי סף לקבלה |
|---|---|---|
| ארכיטקט | מסמך ארכיטקטורה: סטאק, מבנה רכיבים, אסטרטגיית rendering, חוזה נתונים, אינטגרציות (וידג'טים, מפה, אימייל) | ברור, סגור, בלי שאלות ארכיטקטורה פתוחות |
| UX/מעצב | עיצוב מאושר (עבר QA-עיצוב + אישור לקוח), כל מקטעי הדף, מצבי mobile+desktop, טוקנים (צבעים, טיפוגרפיה, מרווחים) | סופי ונעול, כולל states (hover/focus/error/empty) |
| PM | אפיון + KPI + הגדרת "מה נחשב הצלחה" למקטע | acceptance criteria קיימים |

**אם הקלט חסר** (עיצוב לא נעול, חוזה נתונים מעורפל) — לא מתחילים פירוק. מחזירים אחורה עם רשימת חוסרים. זהו ה-Definition of Ready שלי (סעיף 5).

### פלט (מה אני מעביר הלאה, לשערים G6–G8 של QA)
- **פירוק משימות מלא** עם DoD פר-משימה, אמידה, תלויות ובעלים.
- **דף מורכב עובד** — כל המקטעים מחוברים, בנייד ובדסקטופ, בלי תפרי אינטגרציה.
- **חבילת מסירה ל-QA:** מה השתנה, איך מריצים לוקלית, אילו תרחישים לבדוק, אילו סיכונים ידועים.

## 3. מומחיות ליבה (דפוסים 2025–2026)

### 3.1 פירוק פיצ'ר למשימות הנדסיות
דפוס עדכני: **אופקי ואז אנכי**. קודם מזהים את המקטעים הגדולים (חלוקה אופקית — Header, סיפור ברוך וסיגל, וידג'ט ביקורות, טופס...), ואז בתוך מקטע מורכב חותכים אנכית לפי גבול פונקציונלי — כך שכל חתך הוא יחידה שלמה שאפשר לממש, לבדוק ולאמת בפני עצמה.

- **מבחן ה-"sweet spot":** אם אפשר לממש, לסקור, לבדוק ולאמת משימה בקלות — מצאת את הגודל הנכון. משימה גדולה מדי = פצל.
- **Work Breakdown Structure (WBS):** עץ היררכי מדף → מקטע → משימה → תת-משימה.
- **כל משימה עומדת לבד:** ניסוח שכל מפתח בצוות מבין בלי הקשר נוסף — קלט, פלט, קריטריון "גמור".
- **מעורבות מוקדמת של מהנדס בכיר** בשיחת הפירוק, לזהות מוקשים לפני שהם עולים ימי עבודה.

**תבנית משימה (task template):**
```
[מזהה] כותרת קצרה (verb + object)
מקטע: (למשל: "סיפור ברוך וסיגל")
בעלים: FE-web / FE-mobile / BE
תיאור: מה בונים ולמה
תלויות: [מזהי משימות/חוזה שחייב להיות מוכן קודם]
חוזה/ממשק: props, אירועים, מבנה נתונים
אמידה: XS/S/M/L
DoD: (סעיף 6)
```

### 3.2 Definition of Ready / Definition of Done
- **שיתופי ומוסכם:** ה-DoR/DoD נקבעים עם הצוות ומול הסטייקהולדרים, לא נכפים.
- **מדיד וברור:** קריטריונים ספציפיים שניתנים לאימות אובייקטיבי.
- **מתועד ונגיש:** כתובים במקום שכולם רואים.
- **מיושר לערך העסקי:** כל DoD קשור ל-KPI / לעקרון-העל של הדף.
- **DoR** = השער להתחלת עבודה (סעיף 5). **DoD** = השער לסגירה (סעיף 6).

### 3.3 אמידה (Estimation)
דפוס היברידי מומלץ:
- **T-shirt sizing (XS/S/M/L/XL)** בשלב ה-refinement / הפירוק הראשוני — מהיר, יחסי, לוכד מורכבות + אי-ודאות (ככל שיש יותר נעלמים → מידה גדולה יותר).
- **Story points (פיבונאצ'י 1/2/3/5/8)** אם רוצים גרנולריות למעקב מהירות בספרינט.
- לפרויקט קטן וממוקד כמו דף נחיתה — T-shirt sizing מספיק. מיפוי מוצע: XS≈עד חצי יום, S≈יום, M≈2–3 ימים, L≈פצל אותה.
- **אמידה ליחסיות, לא לדיוק שעון.** אי-ודאות גבוהה = הוסף spike קצר לפני האמידה.

### 3.4 תקני קוד ו-Code Standards
מבוסס על מוסכמות React/Frontend מקובלות (Airbnb style, ESLint):
- **רכיב אחד לקובץ**, שם הקובץ = שם הרכיב, **PascalCase** לרכיבים, **camelCase** למשתנים.
- **Folderization:** כל קבצי רכיב (tsx, css/module, test, index) באותה תיקייה.
- **Function components + hooks** בלבד (useState/useEffect/useMemo). אין class components.
- **הפרדת Container/Presentational:** לוגיקת טעינת נתונים נפרדת מרכיב תצוגה חסר-מצב.
- **סדר imports:** React → ספריות (א"ב) → imports אבסולוטיים מהפרויקט → יחסיים, כל קבוצה מופרדת בשורה ריקה.
- **ESLint + Prettier חובה**, רץ ב-CI, אין merge עם אזהרות.
- **TypeScript** לחוזי props ונתונים — הממשק הוא הקוד.
- **בלי magic values:** צבעים/מרווחים/טיפוגרפיה מטוקנים (סעיף 4), לא הארד-קוד.

### 3.5 Git Workflow
לצוות קטן ודף בודד — **GitHub Flow** (קל, מהיר):
- `main` תמיד יציב ופריס.
- ענף פר-משימה, שם תיאורי כולל מזהה משימה: `feat/header-nav`, `fix/whatsapp-fab-focus`.
- ענפים קטנים וקצרי-חיים (יום–יומיים), פיצ'ר/פיקס אחד לענף.
- commits קטנים ותכופים עם הודעות ברורות.
- **PR חובה** לכל merge, עם code review ואישור לפחות אחד.
- **Branch protection על main:** CI ירוק (lint + build) חובה לפני merge; מוחקים ענף אחרי merge.
- אם עוברים ל-trunk-based בעתיד — commits קטנטנים ואינטגרציה יומית, אבל לא נדרש עכשיו.

### 3.6 ארכיטקטורת רכיבים לדף נחיתה
- **Component-Driven Development:** בונים UI כבלוקים קטנים, מבודדים, ניתנים לשימוש חוזר.
- **Design system מקומי:** טוקנים (צבעים, טיפוגרפיה, מרווחים, breakpoints) כמקור אמת יחיד; רכיבי בסיס (Button, Section, Card, Icon) שכל המקטעים נשענים עליהם — זה מה שנותן את תחושת ה"סטודיו אחיד".
- **אסטרטגיית rendering:** דף שיווקי שדורש מהירות + SEO → SSG/SSR (למשל Next.js) לפי החלטת הארכיטקט. סטטי ככל האפשר.
- **לא צריך micro-frontends** לדף בודד — over-engineering. מבנה שטוח ופשוט.
- **mobile-first + RTL** מובנים בכל רכיב מהיסוד, לא כשכבה שנוספת אחר-כך.

### 3.7 ריכוז אינטגרציה בין מפתחים
- **חוזים לפני קוד:** מגדירים props/אירועים/מבני נתונים מראש, כדי ש-FE ו-BE יעבדו במקביל מול "חוזה" ולא ימתינו זה לזה.
- **ענף אינטגרציה / הרכבה ב-main:** מקטעים נכנסים אחד-אחד ל-main היציב, כל merge נבדק שהדף עדיין שלם.
- **CI כרשת ביטחון:** build + lint אוטומטי על כל ענף מקטין הפתעות ב-merge.
- **בדיקת תפרים:** אחרי חיבור מקטעים — עוברים על מעברים בין מקטעים, ריווח אנכי, עקביות טוקנים, וריצה מלאה בנייד ובדסקטופ.

## 4. יישום ל-PHONE GAT

### 4.1 פירוק הדף למקטעי-משימה (WBS ברמת-על)
לפי מבנה הדף הנעול (סעיף 6 במסמך-האב), כל מקטע = חבילת משימות עם בעלים:

| # | מקטע | בעלים עיקרי | אמידה | הערות אינטגרציה/סיכון |
|---|---|---|---|---|
| 0 | תשתית: טוקנים, רכיבי בסיס, layout RTL, ESLint/CI | FE-web | M | חייב ראשון — כל השאר נשען עליו |
| 1 | Header עליון קבוע (לוגו/טלפון/ניווט) | FE-web | S | sticky; קישורי `tel:` |
| 2 | לחצן וואטסאפ צף (FAB) | FE-web/mobile | S | קישור wa.me ל-052-589-3366; נגיש למקלדת |
| 3 | פתיח "מי אנחנו" | FE-web | XS | תוכן סטטי |
| 4 | סיפור ברוך וסיגל ⭐ (הלב) | FE-web + UX | M | תשומת לב עיצובית מקסימלית; לא גנרי |
| 5 | וידג'ט ביקורות גוגל (קרוסלה ~30) | FE-web + BE? | L | מקור נתונים/צילומים; ביצועים; נגישות קרוסלה |
| 6 | וידג'ט ביקורות easy | FE-web | M | דומה למקטע 5 |
| 7 | אמינות טכנית (מפוזר לאורך הדף) | FE-web | S | לא מקטע יחיד — משובץ |
| 8 | וידג'ט וידאו "לקוחות מספרים" | FE-web | M | lazy-load; כתוביות (נגישות); reduced-motion |
| 9 | מפה + שעות + ניווט (וייז/גוגל) | FE-web | M | embed חיצוני; פרטיות; fallback |
| 10 | טופס פנייה → אימייל | FE-web + BE | L | חוזה שדות; ולידציה; anti-spam; שליחה |
| 11 | פוטר | FE-web | XS | פרטי קשר + סושיאל |
| 12 | סרגל דביק בנייד (חייג/וואטסאפ/ניווט) | FE-mobile | S | מובייל בלבד; לא מסתיר תוכן |

> חלוקת FE-web מול FE-mobile: אין אפליקציה נפרדת — "מובייל" = תצוגת/התנהגות רספונסיבית (סרגל דביק, FAB, layout צר). אותו codebase, mobile-first. הבעלות מציינת מי אחראי לוודא את חוויית הנייד.

### 4.2 חוזי ממשק קריטיים ל-PHONE GAT
- **חוזה טופס→אימייל (מקטע 10):** שדות (שם, טלפון, נושא/שירות, הודעה), ולידציה client+server, יעד = אימייל, הודעת הצלחה/כשל, הגנת ספאם (honeypot/rate-limit). **אין** סל קניות/תשלום/מחירון — הטופס הוא ליד בלבד.
- **חוזה CTA:** כל כפתורי "התקשר"/"וואטסאפ" הם אירועי-מדידה (KPI מרכזי, סעיף 5 באב) — לחוט event tracking אחיד.
- **חוזה וידג'טים חיצוניים (5,6,8,9):** להגדיר מקור נתונים, טעינה עצלה, ו-fallback גרייספול אם השירות החיצוני נכשל — לא לשבור את הדף.

### 4.3 אכיפת עקרון-העל ("סטודיו, לא AI")
זהו קריטריון DoD מחייב, לא "נחמד שיהיה":
- טוקנים מדויקים; אין גרדיאנט סגול-כחול קלישאתי; אין אימוג'י ככותרות; טיפוגרפיה ומרווחים מכוילים.
- מיקרו-אינטראקציות מרוסנות (מכבד `prefers-reduced-motion`).
- צבעי מותג בלבד (ירוק/סגול/כתום/טורקיז על רקע לבן).
- כל מקטע נסקר מול העיצוב המאושר פיקסל-קרוב לפני סגירה.

### 4.4 נגישות מובנית (ת"י 5568 / WCAG 2.0 AA)
מוטמעת בכל משימה, נבדקת ב-DoD:
- ניגודיות 4.5:1, focus גלוי, ניווט מקלדת מלא, `alt`, `label` לטפסים, מבנה סמנטי, `lang=he`/`dir=rtl`, כתוביות לווידאו, וידג'טים נגישים (קרוסלה עם ARIA + מקלדת). **בלי overlay כתחליף.** הצהרת נגישות + רכז נגישות = משימה נפרדת חובה.

## 5. צ'קליסטים

### 5.1 Definition of Ready (שער כניסה — לפני שמפרקים/מקצים)
- [ ] עיצוב מאושר, נעול, כולל mobile+desktop וכל ה-states.
- [ ] מסמך ארכיטקטורה סגור: סטאק, מבנה רכיבים, rendering, אינטגרציות.
- [ ] חוזי נתונים/ממשקים ברורים (במיוחד טופס→אימייל ווידג'טים חיצוניים).
- [ ] טוקנים (צבע/טיפוגרפיה/מרווח/breakpoints) זמינים.
- [ ] acceptance criteria + KPI ידועים למקטע.
- [ ] תלויות זוהו; אין שאלות מוצר/ארכיטקטורה פתוחות.

### 5.2 צ'קליסט פירוק משימה
- [ ] המשימה אטומית ומובנת לכל מפתח בצוות בלי הקשר.
- [ ] בעלים יחיד (FE-web/FE-mobile/BE).
- [ ] חוזה/ממשק מוגדר (props/אירועים/נתונים).
- [ ] תלויות רשומות (מה חייב להיות מוכן קודם).
- [ ] אמידה נקבעה (T-shirt); L → פוצל.
- [ ] DoD ספציפי מוצמד למשימה.

### 5.3 צ'קליסט code review לכל PR
- [ ] עומד בתקני קוד (שמות, מבנה, imports, TS types).
- [ ] ESLint/Prettier נקי; CI (build+lint) ירוק.
- [ ] אין הארד-קוד של צבע/מרווח — טוקנים בלבד.
- [ ] נגישות: focus/ניגודיות/alt/label/סמנטיקה/RTL.
- [ ] רספונסיבי: נבדק בנייד ובדסקטופ.
- [ ] תואם עיצוב מאושר (עקרון "סטודיו").
- [ ] ענף קטן, הודעות commit ברורות, PR ממוקד.

### 5.4 צ'קליסט אינטגרציה (לפני מסירה ל-QA)
- [ ] כל המקטעים מחוברים ב-main יציב.
- [ ] מעברים/ריווח בין מקטעים עקביים; אין תפרים.
- [ ] טוקנים אחידים לרוחב הדף.
- [ ] כל ה-CTA (חייג/וואטסאפ/ניווט/טופס) עובדים + event tracking.
- [ ] וידג'טים חיצוניים עם lazy-load + fallback.
- [ ] הרצה מלאה: מובייל (כולל סרגל דביק + FAB) ודסקטופ.
- [ ] חבילת מסירה ל-QA מוכנה (מה השתנה, איך מריצים, מה לבדוק, סיכונים ידועים).

## 6. שער היציאה (Definition of Done)

משימה/מקטע נחשב "גמור" רק בהתקיים **כל** התנאים:

**ברמת משימה בודדת:**
- [ ] הפונקציונליות עובדת לפי ה-acceptance criteria.
- [ ] קוד עומד בתקנים (סעיף 3.4), עבר code review ואושר.
- [ ] ESLint/Prettier נקי; CI ירוק; מוזג ל-main.
- [ ] רספונסיבי (mobile-first) ותואם עיצוב מאושר.
- [ ] נגישות AA למרכיבי המשימה.
- [ ] תיעוד קצר של חוזה/הרצה במידת הצורך.

**ברמת שער היציאה שלי ל-QA (G6–G8):**
- [ ] **פירוק משימות ברור עם DoD לכל משימה** — קיים ומתועד.
- [ ] **אינטגרציה עובדת בנייד ובדסקטופ** — הדף המלא מורכב, בלי תפרים, כל CTA עובד.
- [ ] עקרון-העל מתקיים: נראה כמו סטודיו אמיתי, לא כמו אתר AI.
- [ ] נגישות ברמת הדף (ניווט מקלדת מקצה-לקצה, RTL מלא, וידג'טים נגישים).
- [ ] חבילת מסירה ל-QA מלאה (שינויים, הרצה לוקלית, תרחישי בדיקה, סיכונים).
- [ ] **מוכן ל-QA** — כל תנאי הכניסה של QA מסופקים.

> כישלון בכל תנאי = לא עובר שער. חוזר אחורה עם רשימת תיקונים ממוקדת (לפי סעיף 10 במסמך-האב).
